# FutureMirror 技術構成レビュー：強みと改善点

このドキュメントは、プロジェクト完成後に技術構成を振り返り、「何が良かったか」「何を変えるべきか」「次に活かせることは何か」を整理したものである。

---

## 技術構成の全体像

| レイヤー | 採用技術 | 備考 |
|---------|---------|------|
| バックエンド | Laravel 12 (PHP 8.2+) | フルスタックフレームワーク |
| フロントエンド | Blade テンプレート + Vanilla JS | SPA ではなくサーバーサイド描画 |
| スタイリング | Tailwind CSS v4 | Vite 経由 + CDN の二重構成 |
| データベース | SQLite | ファイルベースDB |
| ビルドツール | Vite 6 | Laravel Vite Plugin で連携 |
| 認証 | Laravel Breeze | 標準の認証スターターキット |
| HTTP通信 | axios | CSRF トークン付き |

## 強み（良かった点）

### 1. ハッカソン向きのシンプルな技術選定
- **SQLite + Laravel** の組み合わせは、DB サーバーの構築が不要で、`php artisan migrate` だけですぐ動く。チーム全員が同じ手順で環境構築でき、ハッカソンのように時間が限られる場面では大きな強みになる。
- 学び：**限られた時間では「すぐ動く」ことが最優先。構成をシンプルにする判断は正しかった。**

### 2. Blade テンプレートによる開発速度
- React や Vue のような SPA を導入しなかったことで、フロントエンドのビルド設定やAPI 設計にかける時間を削減できた。
- Controller でデータを取得し、そのまま Blade に渡して表示する流れは直感的で、バックエンドとフロントエンドを行き来する回数が少ない。
- 学び：**画面数が少なく、リアルタイム更新が不要なアプリなら、Blade で十分に戦える。SPA は必要になってから導入すればよい。**

### 3. マイルストーン自動生成の設計判断
- ユーザーが目標を入力するだけで、カテゴリーに応じた4つのマイルストーンが自動生成される仕組みは、MVP として「使ってもらえる最小限」を実現するうえで効果的だった。
- 学び：**「ユーザーの入力を減らす」設計は体験を良くする。自動生成できる部分は自動化した方がよい。**

### 4. 複数の可視化方式（フロー + WBS）
- 同じデータを「縦のフロー表示」と「横のタイムライン表示」の2通りで見せることで、ユーザーが自分に合った見方を選べる。機能としての差別化にもなっている。
- 学び：**データの見せ方を複数用意することは、コードの追加量に対してユーザー価値が大きい。**

### 5. Tailwind CSS による高速なUI構築
- ユーティリティクラスを HTML に直接書く方式は、デザインの試行錯誤が速い。CSS ファイルとの往復が不要で、画面を見ながらクラスを足していくだけで見た目が整う。
- カスタムカラーテーマ（primary: 緑、secondary: 青、accent: ピンク）を定義したことで、デザインの一貫性も保てた。
- 学び：**Tailwind はプロトタイプ開発と相性が良い。カラーテーマを最初に決めておくとブレない。**

### 6. Breeze による認証の即時導入
- ログイン、ユーザー登録、パスワードリセットをゼロから作らず Breeze で一括導入したのは、時間の使い方として正解だった。認証は自作するとバグやセキュリティリスクが高い。
- 学び：**認証は自作しない。フレームワークが提供するものを使うのが鉄則。**

## 改善点（次回に活かすべき点）

### 1. データベース設計の堅牢性（優先度：高）

**現状の問題：**
- 外部キー制約がコメントアウトされており、DB レベルでデータの整合性が保証されていない。
- goal 作成時にトランザクションを使っていないため、途中失敗で goal だけ残り milestone がない状態が起こりうる。
- user_id、goal_id にインデックスがなく、データ増加時に検索が遅くなる。

**次回やるべきこと：**
```
1. 外部キー制約を有効にする（migration で $table->foreign() を使う）
2. 複数テーブルへの保存は DB::transaction() で囲む
3. 検索に使うカラムにインデックスを追加する
4. Model に hasMany / belongsTo のリレーションを定義する
5. casts で日付・真偽値の型を明示する
```

**学び：ハッカソンでは動くことを優先して制約を省くのは仕方ないが、「後で直す」リストを最初から作っておくと、改善フェーズに入りやすい。**

### 2. フロントエンドの JavaScript 管理（優先度：高）

**現状の問題：**
- 各 Blade ファイルに `<script>` タグで数百行の JavaScript が直接書かれており、画面間でロジックが重複している。
- `onclick="..."` 属性による処理は見通しが良い反面、処理が HTML 中に散在して保守しにくい。
- 共通の JavaScript（resources/js/app.js）がほぼ空で、画面横断の共通処理が存在しない。

**次回やるべきこと：**
```
1. 共通処理（ボトムシート制御、トースト通知、チェックボックス切替）を
   resources/js/ 配下の個別ファイルに切り出す
2. onclick 属性の代わりに addEventListener でイベントを登録する
3. 画面固有のロジックも Blade 内の <script> ではなく
   resources/js/pages/ のような構成に移す
4. 将来的に画面が増えるなら Alpine.js や Livewire の導入も検討する
```

**学び：Blade 内に JS を書くのは初速が速いが、3画面を超えると管理が破綻し始める。「共通化のタイミング」を意識しておくとよい。**

### 3. CSS の管理方法の統一（優先度：中）

**現状の問題：**
- Tailwind CSS が Vite 経由（ビルド管理）と CDN（直接読み込み）の2系統で読み込まれている。
- Blade ファイル内に `<style>` タグでインラインスタイルが書かれている画面もある。
- どこでスタイルが定義されているか追いにくい。

**次回やるべきこと：**
```
1. CDN を廃止し、Vite 経由の Tailwind に一本化する
2. カスタムテーマの定義を resources/css/app.css の @theme に集約する
3. インラインスタイルは Tailwind のユーティリティクラスに置き換える
4. どうしてもカスタム CSS が必要な場合は @layer components に書く
```

**学び：「どこにスタイルがあるか」が分散すると、デザイン変更のたびに複数箇所を探す羽目になる。1箇所に寄せるルールを最初に決めるとよい。**

### 4. カラム命名の一貫性（優先度：中）

**現状の問題：**
- milestones テーブルの `startDate`、`endDate` が camelCase で、Laravel の慣例（snake_case: `start_date`、`end_date`）と異なる。
- 今後 Eloquent のリレーションやアクセサを使う際に不便が生じる。

**次回やるべきこと：**
```
1. 新規プロジェクトでは最初から snake_case で統一する
2. 既存プロジェクトではマイグレーションでカラム名を変更するか、
   Model の $appends と accessor で吸収する
```

**学び：命名規則はプロジェクト初日に決めておくべき。フレームワークの慣例に合わせるのが最もトラブルが少ない。**

### 5. Breeze 生成ファイルの整理（優先度：低）

**現状の問題：**
- Breeze が自動生成したログイン画面（auth/login.blade.php）と、独自に作ったログイン画面（login.blade.php）の両方がプロジェクトに存在する。
- 使われていない Breeze の画面ファイルが残っており、初見のメンバーがどちらが本番か迷う。

**次回やるべきこと：**
```
1. 使わない Breeze 画面は削除するか、明示的にコメントで「未使用」と記す
2. 独自画面と Breeze 画面の役割分担をREADMEに書く
```

**学び：自動生成されたコードのうち使わない部分は、早めに整理しないと混乱の元になる。**

## 技術選定の判断基準（次回のために）

今回の経験から、技術選定時に使える判断軸を整理する。

### いつ SPA（React/Vue）を選ぶべきか
- リアルタイム更新が必要（チャット、コラボレーション機能など）
- 画面遷移なしでの複雑な状態管理がある
- API を他のクライアント（モバイルアプリなど）と共有したい
- 上記に当てはまらないなら、Blade で十分

### いつ SQLite 以外のDBを選ぶべきか
- 複数サーバーから同時にDBアクセスする場合（SQLite はファイルロック）
- 大量の同時書き込みがある場合
- 本番運用でバックアップや複製が必要な場合
- ハッカソンや個人開発の初期段階なら SQLite で問題ない

### いつフロントエンドフレームワークを入れるべきか
- 画面内の状態が複雑になってきたら（5つ以上の状態変数を手動で管理している）
- 同じUIパターンを3画面以上で使い回しているなら
- 最初から入れるのではなく、「つらくなったら」が導入のタイミング

## 全体の一言まとめ

このプロジェクトの技術構成は、**ハッカソンという制約の中で「動くものを速く作る」という目的に対して適切だった**。SQLite、Blade、Tailwind の組み合わせは初速が速く、4画面のアプリを短期間で作り上げるのに十分な選択だった。

一方で、**開発が進むにつれて「構造を整理するコスト」が増えていく構成でもある**。JavaScript の分散、CSS の二重管理、DB制約の不足は、今の規模なら問題にならないが、機能追加や他の人との共同開発が始まると顕在化する。

次回のプロジェクトでは、「初速を保ちつつ、あとで困らない最低限の構造」を意識すること。具体的には：
1. **DB設計は最初から外部キー制約とリレーションを入れる**（コストはほぼゼロ）
2. **JavaScript は3画面を超えたら共通ファイルに切り出す**（タイミングを決めておく）
3. **CSS の読み込み方法は1つに統一する**（ルールを初日に決める）
4. **命名規則はフレームワークの慣例に従う**（snake_case で統一）
