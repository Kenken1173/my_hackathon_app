# 初学者向け解説スキル

## 目的
- ユーザーが「解説して」と言ったとき、指定された対象を初学者でも理解できる形で解説する。
- 読んだ後に「自分で人に説明できる」「自分で実装できる」状態になることをゴールとする。
- 毎回複雑なプロンプトを書かなくても、一定の品質とフォーマットで解説が出力される。

## トリガー
- 「解説して」「説明して」と言われたとき。
- コードや設定ファイルを選択した状態で解説を求められたとき。
- プロジェクト構成やアーキテクチャの説明を求められたとき。

## 出力フォーマット

以下の見出し構成を必ず守る。内容がないセクションは省略してよい。

```markdown
## [解説対象のタイトル]

### まず知っておきたい用語
- **用語名**：1〜2 文で平易に説明する。例え話や具体例があるとなお良い。
- （解説本文で使う専門用語をすべてここで先に説明する）

### 調べ方・考え方の手順
- この対象を自分で調べるとしたら、どういう順番で何を見ればよいかをステップで書く。
- 各ステップで見るファイルや確認する内容を具体的に書く。
- ファイル名や設定名が出てきたら、カッコ書きでそれが何かを補足する。

### このプロジェクトで分かったこと
- 実際に調べて判明した事実を書く。
- 内容が多い場合は #### 小見出しでトピックごとに分ける。
- 箇条書きの中でも、初出の用語にはカッコ書きで補足を入れる。
- 「つまり〜」で要約を挟み、ここまでの話を整理する。

### 注意点・改善のヒント
- 現状の問題点や将来困りそうなポイントを書く。
- なぜ問題なのかの理由、具体的にどう改善するかの方法もセットで書く。

### 一言まとめ
- セクション全体を 1〜2 文で総括する。
```

## 解説の書き方ルール

### 用語の扱い
1. 解説本文で使うすべての専門用語は「まず知っておきたい用語」セクションで先に定義する。
2. 用語定義は太字（`**用語**`）＋コロン＋平易な説明の形式にする。
3. 本文中で初出の用語にはカッコ書きで短い補足を付ける（例：「routes/web.php（URL と処理の対応を書くファイル）」）。
4. 略語は正式名称を併記する（例：「CDN（Content Delivery Network）」）。
5. 対義語や関連概念がある場合は「対義語は〜」「〜と似ているが違う点は〜」のように関係を示す。

### 説明の粒度
1. 「なぜそうなっているか」「それがないとどう困るか」を添える。定義だけでなく意味を伝える。
2. 抽象的な説明のあとに「つまり〜」「例えば〜」で具体化する。
3. 比較対象がある場合は対比で説明する（例：「SPA ではなく SSR」「CDN ではなく Vite 経由」）。
4. コードや設定値の例は最小限のインラインコードで示す（例：`text-red-500`、`onclick="..."`）。

### 構成の組み立て方
1. 対象のコードや設定をまず読み、関連ファイルも必要に応じて確認してから書き始める。
2. ユーザーが選択した範囲だけでなく、理解に必要な周辺コンテキスト（呼び出し元、設定ファイル、ルーティングなど）も調べた上で解説する。
3. 「調べ方・考え方の手順」は、次に似た状況に出会ったときにユーザーが自力で調べられるようにする目的で書く。
4. 「分かったこと」は事実の羅列ではなく、トピックごとにまとめて流れを作る。

### やってはいけないこと
- 用語を説明なしで使うこと。
- 「〜が重要です」だけで、なぜ重要かを書かないこと。
- 本文中に用語集で定義していない専門用語が突然出ること。
- フォーマットの見出し構成を勝手に変えること。
- 解説の前後に不要な前置きや締めの挨拶を入れること。

## 出力先
- ユーザーから特に指定がなければ、チャット上に直接出力する。
- 「ファイルに書いて」「learnings に追記して」等の指示があれば、指定されたファイルに追記する。
